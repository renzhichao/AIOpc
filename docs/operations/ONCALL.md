## 1. 值班概述

### 1.1 目的

建立 7x24 小时事故响应能力，确保生产环境问题得到及时处理，最小化服务中断时间。

### 1.2 适用范围

- **生产环境服务器**: 118.25.0.190 (Platform), 101.34.254.52 (Remote Agent)
- **服务组件**: Backend, Frontend, PostgreSQL, Redis
- **监控告警**: P0/P1 级别告警

### 1.3 值班原则

- **快速响应**: P0 告警 15 分钟内响应
- **有效处理**: 遵循事故响应流程，系统性解决问题
- **充分沟通**: 及时更新状态，保持团队信息同步
- **持续改进**: 每次事故后复盘，改进流程

## 2. 值班轮换

### 2.1 值班结构

```yaml
主值班工程师 (Primary):
  轮换周期: 1 周
  交接时间: 每周一 00:00 (北京时间)
  职责: 第一响应人，处理所有告警
  要求: 全天候保持通讯畅通

备用值班工程师 (Secondary):
  轮换周期: 与 Primary 同步
  职责: Primary 备份，辅助处理事故
  要求: Primary 无法响应时接手
```

### 2.2 值班表示例

| 周次 | 开始日期 | 主值班工程师 | 备用值班工程师 | 状态 |
|------|---------|-------------|---------------|------|
| 第 1 周 | 2026-03-18 | @ engineer-1 | @ engineer-2 | 🟢 当前周 |
| 第 2 周 | 2026-03-25 | @ engineer-2 | @ engineer-3 | ⏳ 下一周 |
| 第 3 周 | 2026-04-01 | @ engineer-3 | @ engineer-1 | ⏳ 计划中 |
| 第 4 周 | 2026-04-08 | @ engineer-1 | @ engineer-2 | ⏳ 计划中 |

### 2.3 值班工程师要求

**必备技能**:
- 熟悉系统架构和组件
- 掌握常见故障排查方法
- 能够独立处理 P1/P2 级别事故

**通讯要求**:
- 钉钉消息 15 分钟内回复
- 电话呼叫立即接听
- 钉钉群 @ 提醒及时响应

## 3. 值班职责

### 3.1 主值班工程师职责

**告警监控**:
- 实时监控钉钉告警消息
- 定期检查监控 Dashboard（每日至少 3 次）
- 关注系统健康状态

**事故响应**:
- P0: 15 分钟内响应并开始处理
- P1: 1 小时内响应并开始处理
- P2: 24 小时内处理

**沟通协调**:
- 在 #incidents 群及时更新状态
- 每 30 分钟更新一次事故进展
- 必要时升级给 Tech Lead

**文档记录**:
- 记录所有处理的事故
- P0/P1 事故编写复盘报告
- 更新 Runbook 和故障排查指南

### 3.2 备用值班工程师职责

**备份支持**:
- Primary 无法响应时接手（30 分钟无响应）
- 辅助 Primary 处理复杂事故
- 参与 P0/P1 事故处理

**学习成长**:
- 观察 Primary 处理事故的方法
- 学习系统架构和故障排查
- 准备未来担任 Primary

### 3.3 值班免责

以下情况不需要值班工程师处理：
- 计划内维护（提前通知的）
- 非生产环境问题
- 第三方服务故障（Feishu、DeepSeek API）
- 误报告警（需标记并优化规则）

## 4. 响应时间要求

### 4.1 响应时间定义

| 严重级别 | 响应时间 | 处理时间目标 | 更新频率 |
|---------|---------|-------------|---------|
| **P0 - 严重** | 15 分钟 | MTTR < 1 小时 | 每 15 分钟 |
| **P1 - 高** | 1 小时 | MTTR < 4 小时 | 每 30 分钟 |
| **P2 - 中** | 24 小时 | MTTR < 2 天 | 每日更新 |

### 4.2 严重级别定义

#### P0 - 严重 (Critical)

**定义**: 完全服务中断或核心功能不可用

**示例**:
- 所有用户无法访问服务
- 数据库完全不可用
- OAuth 登录 100% 失败
- 数据丢失或损坏
- 服务器宕机

**响应要求**:
- ⚠️ 15 分钟内响应
- 📞 立即电话通知 Primary
- 🔄 升级路径: Primary → Tech Lead (30 min) → CTO (1 hour)

#### P1 - 高 (High)

**定义**: 主要功能故障，部分用户受影响

**示例**:
- 性能严重下降（P95 > 3 秒）
- 单个功能完全不可用
- OAuth 登录部分失败
- 错误率 > 10%
- 容器 unhealthy 状态

**响应要求**:
- ⏰ 1 小时内响应
- 💬 钉钉消息通知
- 🔄 升级路径: Primary → Tech Lead (2 hours)

#### P2 - 中 (Medium)

**定义**: 次要功能故障，少数用户受影响

**示例**:
- 非核心功能不可用
- UI 显示问题
- 文档错误
- 性能轻微下降

**响应要求**:
- 📅 24 小时内处理
- 📝 创建 GitHub Issue 跟踪
- 🔄 无需升级

## 5. 升级路径

### 5.1 升级策略

```yaml
Level 1: On-Call 工程师 (Primary)
  响应时间: P0 < 15min, P1 < 1hour
  处理时间: P0 < 30min 初步诊断
  升级触发: 30 分钟内未响应 OR 1 小时未解决

Level 2: Tech Lead
  升级条件: Primary 30 分钟未响应 OR P0 持续 1 小时未解决
  决策权限: 可以执行任何必要操作恢复服务
  升级触发: P0 持续 2 小时未解决 OR 需要额外资源

Level 3: CTO
  升级条件: Tech Lead 30 分钟未响应 OR 重大决策需求
  决策权限: 最高决策权，可以协调所有资源
  无升级: 这是最终升级级别
```

### 5.2 升级流程

**何时升级**:
- Primary 30 分钟内未响应告警
- Primary 1 小时内无法解决 P0 事故
- 需要额外资源或权限
- 事故影响范围扩大

**如何升级**:
1. **钉钉群升级**: 在 #on-call 群 @ Tech Lead
2. **电话升级**: 钉钉消息 5 分钟无回应，直接电话
3. **记录升级**: 在事故记录中标记升级时间和原因

**升级后**:
- 原值班工程师继续协助
- 升级人员主导事故处理
- 保持团队信息同步

## 6. 交接流程

### 6.1 交接时间

- **常规交接**: 每周一 00:00
- **特殊交接**: 事故处理中延迟交接，直到解决

### 6.2 交接清单

**当前状态**:
- [ ] 当前是否有活跃事故？
- [ ] 当前是否有待处理告警？
- [ ] 系统是否有已知问题？

**待办事项**:
- [ ] 待处理的 GitHub Issues
- [ ] 待跟进的告警调优
- [ ] 待更新的文档

**权限和访问**:
- [ ] 监控 Dashboard 访问
- [ ] 服务器 SSH 访问
- [ ] 钉钉群成员身份
- [ ] GitHub Issue 权限

### 6.3 交接步骤

1. **交接准备** (交接前 1 小时):
   - Primary 整理当前状态
   - 确认无活跃事故
   - 准备交接信息

2. **交接执行** (00:00):
   - Primary 在 #on-call 群宣布交接
   - 新 Primary 确认接班
   - 查看 Dashboard 确认系统健康

3. **交接确认** (00:15):
   - 新 Primary 查看监控面板
   - 确认所有权限已配置
   - 在 #on-call 群确认交接完成

**交接模板**:
```
🔄 值班交接

时间: YYYY-MM-DD HH:00
交出: @ engineer-1
接手: @ engineer-2

系统状态: ✅ 正常
活跃事故: 无
待办事项: Issue #123 需跟进

交接完成，@ engineer-2 开始值班
```

## 7. 沟通渠道

### 7.1 钉钉群组

#### #on-call 群
- **用途**: 值班沟通和交接
- **成员**: 所有工程师
- **机器人**: 告警通知机器人
- **消息保留**: 90 天

#### #incidents 群
- **用途**: 事故响应协调
- **成员**: 所有工程师 + Tech Lead + CTO
- **机器人**: 告警通知机器人
- **消息保留**: 永久

### 7.2 告警路由

```yaml
P0 告警:
  渠道: 钉钉群 + 电话
  频率: 每 5 分钟重发直到确认
  确认: 需要手动确认收到

P1 告警:
  渠道: 钉钉群
  频率: 每 15 分钟重发直到确认
  确认: 需要手动确认收到

P2 告警:
  渠道: 钉钉群
  频率: 单次发送
  确认: 不需要确认
```

### 7.3 事故沟通

**内部沟通** (#incidents 群):
- 每 30 分钟更新一次
- 使用标准更新模板
- 标记严重级别和影响范围

**外部沟通** (P0 > 15 分钟):
- 状态页面: status.aiopc.com (未来)
- 应用内横幅通知
- 社交媒体公告 (重大事故)

## 8. 事故响应流程

### 8.1 响应流程

```mermaid
检测 → 分诊 → 响应 → 解决 → 复盘
```

#### 1. 检测 (Detection)

**来源**:
- 自动化告警（Prometheus + Alertmanager）
- 用户报告（客服、邮件）
- 内部观察（手动检查）

#### 2. 分诊 (Triage)

**确认事故**:
- 验证告警真实性
- 确定严重级别（P0/P1/P2）
- 评估影响范围

**创建事故记录**:
```bash
# 创建 GitHub Issue
标签: incident, P0
标题: [P0] Incident - 简短描述
模板: 使用事故模板
```

**通知值班工程师**:
- 钉钉 @ Primary
- P0 级别附加电话通知

#### 3. 响应 (Response)

**初步诊断**:
- 收集上下文（日志、指标、追踪）
- 确定可能原因
- 评估影响范围

**临时修复**:
- 如果可能，实施快速修复
- 如果不行，准备缓解措施
- 必要时升级

**持续更新**:
- 每 30 分钟更新状态
- 使用标准更新模板
- 保持团队信息同步

#### 4. 解决 (Resolution)

**永久修复**:
- 实施永久解决方案
- 验证问题已解决
- 监控 1 小时防止复发

**关闭事故**:
- 确认服务完全恢复
- 更新事故记录
- 通知团队解决

#### 5. 复盘 (Post-Incident)

**P0/P1 事故必须复盘**:
- 48 小时内安排复盘会议
- 编写复盘报告
- 制定改进行动项

---

### 8.2 更新模板

**初始更新**:
```
🔴 P0 事故开始

标题: {简要描述}
影响: {受影响用户/功能}
开始时间: {YYYY-MM-DD HH:mm}
当前状态: 正在调查
更新时间: {YYYY-MM-DD HH:mm}
```

**进展更新**:
```
📊 事故进展

标题: {简要描述}

进展: {具体进展}
预计恢复: {未知/具体时间}
更新时间: {YYYY-MM-DD HH:mm}
```

**解决更新**:
```
✅ 事故解决

标题: {简要描述}
解决时间: {YYYY-MM-DD HH:mm}
恢复时间: {YYYY-MM-DD HH:mm}
根本原因: {简要描述}
复盘计划: {YYYY-MM-DD}
```

## 9. 值班准备

### 9.1 值班前准备

**第一周值班**（新工程师）:
- [ ] 完成值班培训
- [ ] 跟随资深工程师 shadow 1 周
- [ ] 完成 3 次模拟事故演练
- [ ] 通过值班认证考试

**常规值班**（每周）:
- [ ] 确认通讯工具正常
- [ ] 确认监控权限正常
- [ ] 确认服务器访问正常
- [ ] 查看近期事故和 Runbook

### 9.2 值班期间

**日常工作**:
- 实时监控钉钉告警
- 每日至少 3 次检查 Dashboard
- 及时响应所有 P0/P1 告警
- 更新值班日志

**值班日志**:
- 记录所有处理的事故
- 记录所有告警（包括误报）
- 记录系统观察和发现

### 9.3 值班后

**交接准备**:
- 整理本周处理的事故
- 总结待办事项
- 准备交接信息

**个人恢复**:
- 值班后 1 天不安排重要工作
- 充分休息，避免疲劳累积
- 参加值班回顾会议

## 10. 事后复盘

### 10.1 复盘目的

- 学习和改进
- 预防同类事故
- 优化系统和流程
- 建立无责文化

### 10.2 复盘要求

**必须复盘的事故**:
- 所有 P0 事故
- 导致数据丢失的 P1 事故
- MTTR > 2 小时的 P1 事故

**复盘时间线**:
- Day 1: 事故解决
- Day 2-3: 调查和起草复盘报告
- Day 4: 团队评审和批准
- Day 5: 发布和分享学习

### 10.3 复盘报告模板

```markdown
# Incident Postmortem: {事故标题}

Date: {YYYY-MM-DD}
Severity: P{level}
Incident Commander: {name}
Duration: {start_time} - {end_time} ({duration})

## Summary
{What happened in 1-2 sentences}

## Impact
- Who was affected: {users/regions/features}
- How severe: {error rate, downtime duration}
- Business impact: {revenue, user trust}

## Timeline
- {timestamp}: {event}
- {timestamp}: {event}
...

## Root Cause
{5 Whys analysis}

### Five Whys
1. Why did this happen?
   {answer}
2. Why did {that} happen?
   {answer}
3. ...
4. ...
5. ...

## Resolution
{How did we fix it}

## Follow-up Actions
- [ ] {action} (Who, When)
- [ ] {action} (Who, When)
...

## Learnings
{What did we learn? What will we do differently?}

## Appendix
- Logs: {links}
- Metrics: {links}
- Screenshots: {links}
```

### 10.4 无责原则

- **关注系统，不关注个人**
- **"不要将可以用错误解释的事情归因于恶意"**
- **行动项优先于惩罚**
- **公开分享学习**
